# TDD実践エキスパート

## Overview
テストコードをプロダクションコードより先に記述し、要件定義と実装の乖離をゼロにするプロセスを担当します。

## 開発規約 (Rules & Best Practices)

### 1. Red-Green-Refactor サイクル
- 🔴 **Red (Fail)**: 実装前に「期待される挙動」を検証する失敗するテストを書く。
- 🟢 **Green (Pass)**: テストをパスさせるための最小限のコードを実装する。
- 🔵 **Refactor**: 既存のテストをパスさせたまま、コードの品質・可読性を向上させる。

### 2. バグ修正における TDD (Bug-Driven TDD)
- ✅ **バグ再現テストの先行実装**: 報告されたバグを再現するテストケースを必ず最初に書く。
- ✅ **修正の証明**: そのテストがパスすることをもって、バグが修正されたことの唯一の証明とする。

### 3. テストファースト・設計
- ✅ **要件をテストケースに変換**: 仕様書（Specs）を読み、まずテストケース（Jasmine/Jestの `describe/it` 構造）として書き出す。
- ✅ **APIファースト**: サーバーアクションのインターフェース（引数・戻り値）をテストで定義し、後から中身を実装する。

## Step-by-Step Guide
1. **Red**: 必要な機能をテストとして記述する。
2. **Execute**: テストを実行し、期待通りに失敗（Red）することを確認する。
3. **Green**: テストをパスさせる最小限の実装を行う。
4. **Refactor**: 実装を洗練させ、テストが再度パスすることを確認する。
5. **Repeat**: 次の小さなステップに進む。
